# Industrial Viewer Architecture Guide

This document explains the folder structure of the application, what each important file does, and why it exists.

## High-Level Overview

The project is split into two main parts:

- `frontend app`: the React UI that lets you explore an OPC UA-style node tree
- `demo OPC UA server`: a separate Node.js container that exposes a sample OPC UA endpoint

This separation is intentional. It keeps the viewer independent from the server so you can later replace the demo server with a real OPC UA server or add a proper live backend connector.

## Root Folder

### `/package.json`

Defines the frontend app metadata, dependencies, and scripts.

Why it exists:

- tells Node/npm what packages the frontend needs
- defines commands like `dev`, `build`, and `preview`
- is used by Docker during image build

### `/tsconfig.json`

Top-level TypeScript reference file.

Why it exists:

- connects the app TypeScript config and the Vite/node TypeScript config
- keeps TypeScript setup clean and modular

### `/tsconfig.app.json`

TypeScript settings for the React frontend code in `src/`.

Why it exists:

- controls strict typing for application code
- defines the `@/` import alias for cleaner imports

### `/tsconfig.node.json`

TypeScript settings for build/config files such as `vite.config.ts`.

Why it exists:

- separates app code rules from tooling/config code rules

### `/vite.config.ts`

Vite configuration for the frontend.

Why it exists:

- enables the React plugin
- defines path aliases like `@/components/...`
- controls how the frontend is bundled

### `/tailwind.config.ts`

Tailwind CSS configuration.

Why it exists:

- defines which files Tailwind scans for classes
- extends the theme with custom colors, fonts, shadows, and backgrounds
- creates the visual design system for the app

### `/postcss.config.js`

PostCSS configuration used by Tailwind.

Why it exists:

- allows Tailwind and autoprefixer to process CSS correctly

### `/eslint.config.js`

Linting rules for the frontend code.

Why it exists:

- helps catch errors and bad patterns early
- enforces cleaner TypeScript and React code

### `/index.html`

The single HTML entry point for the frontend.

Why it exists:

- gives Vite a root page to mount the React app into
- contains the root DOM node: `#root`

### `/.gitignore`

Lists files Git should ignore.

Why it exists:

- avoids committing generated output like `node_modules` and `dist`

### `/README.md`

Project quick-start and run guide.

Why it exists:

- gives the basic usage instructions
- explains how to run the app with Docker

### `/ARCHITECTURE.md`

This document.

Why it exists:

- provides a structural guide for understanding and extending the project

## Frontend Source

All frontend application code lives in:

- `/src`

### `/src/main.tsx`

Frontend bootstrap file.

Why it exists:

- starts React
- mounts the `App` component into the HTML page

### `/src/App.tsx`

Main application UI and page composition.

Why it exists:

- contains the overall layout
- manages the main viewer state such as:
  - selected node
  - search query
  - expanded tree items
  - demo mode vs live mode
- renders the main panels:
  - hero/header
  - namespace tree
  - node details
  - related nodes

This is currently the central app file because the project is still in its first version. As the app grows, this file can be split into feature-specific components.

### `/src/index.css`

Global CSS and Tailwind layers.

Why it exists:

- defines the app theme variables
- sets the dark industrial visual style
- imports fonts
- applies base styling to the entire app

### `/src/data/opcua-demo.ts`

Static demo OPC UA data model.

Why it exists:

- contains the built-in example node tree
- powers the demo mode without requiring a live OPC UA server
- lets users learn OPC UA concepts from realistic sample nodes

This file is effectively the current mock backend for the UI.

### `/src/lib/utils.ts`

Shared utility helpers.

Why it exists:

- currently provides `cn(...)`, a helper for merging Tailwind class names
- keeps small reusable helpers out of UI files

## UI Components

Reusable UI primitives live in:

- `/src/components/ui`

These are inspired by the shadcn/ui style, but implemented directly in the repo instead of being generated by the CLI.

Why this approach was used:

- the local machine did not have npm available outside containers
- this keeps the design system self-contained
- it avoids adding unnecessary generator output

### `/src/components/ui/button.tsx`

Reusable button component with variants.

Why it exists:

- keeps button styles consistent across the app
- supports multiple visual styles like primary and secondary

### `/src/components/ui/card.tsx`

Reusable card layout components.

Why it exists:

- standardizes content containers such as header panels and detail panels

### `/src/components/ui/input.tsx`

Reusable text input component.

Why it exists:

- powers search fields and future forms
- keeps form styling consistent

### `/src/components/ui/badge.tsx`

Small tag/label component.

Why it exists:

- shows node class, datatype, status, and other metadata visually

### `/src/components/ui/separator.tsx`

Reusable divider component.

Why it exists:

- visually separates sections in detail panels

### `/src/components/ui/scroll-area.tsx`

Reusable scrollable container.

Why it exists:

- gives the tree panel a styled scroll area
- keeps scrolling behavior consistent

## Docker and Deployment

### `/Dockerfile`

Frontend production image definition.

Why it exists:

- builds the React app inside a Node container
- then copies the built static files into an Nginx container
- creates a small production-ready frontend image

This is a multi-stage build:

1. build stage with Node and npm
2. runtime stage with Nginx

### `/docker/nginx.conf`

Nginx configuration for the frontend container.

Why it exists:

- serves the built frontend files
- routes all paths back to `index.html` so client-side routing works if added later

### `/docker-compose.yml`

Multi-container orchestration file.

Why it exists:

- starts the frontend viewer container
- starts the demo OPC UA server container
- makes local testing simple with one command

Current services:

- `industrial-viewer`
- `opcua-demo-server`

## Demo OPC UA Server

The demo server code lives in:

- `/server-demo`

This is separate from the frontend because it represents a data source, not part of the UI.

### `/server-demo/package.json`

Package definition for the OPC UA demo server.

Why it exists:

- defines the server dependency on `node-opcua`
- defines the `start` script

### `/server-demo/Dockerfile`

Container build file for the demo OPC UA server.

Why it exists:

- installs server dependencies
- starts the OPC UA server in its own container

### `/server-demo/server.js`

The actual demo OPC UA server implementation.

Why it exists:

- creates an OPC UA server on port `4840`
- exposes a simple industrial address space
- defines example objects, variables, and methods
- simulates runtime changes such as temperature and pressure

This gives you a real OPC UA endpoint to connect to later.

## Architectural Intent

The current project structure is designed to support three stages:

### Stage 1: learning and demo

Already implemented.

- frontend demo explorer
- mock namespace in the UI
- standalone demo OPC UA server

### Stage 2: live browsing

Planned next.

- add a backend service that connects to a real OPC UA endpoint
- browse nodes dynamically instead of using only static demo data
- reuse the same frontend panels and components

### Stage 3: industrial deployment

Possible future direction.

- authentication and connection profiles
- multiple OPC UA endpoints
- live subscriptions
- alarms, trends, and historical browsing

## Simple Mental Model

If you want a practical way to think about the folders:

- `src/` = what the user sees
- `src/data/` = demo data the UI reads
- `src/components/ui/` = reusable building blocks
- `server-demo/` = fake industrial system for local OPC UA testing
- `docker-compose.yml` = starts the whole environment
- `Dockerfile` = packages the frontend for container deployment

## If You Want To Extend It

The next clean additions would be:

1. add a `src/features/` folder and move viewer logic out of `App.tsx`
2. add a backend service folder such as `/server-api`
3. connect the frontend to live browse/read endpoints
4. add a Docker dev workflow with hot reload

That would keep the structure scalable without changing the current foundations.
